# 提示词缓存
Prompt Caching（提示词缓存 / 前缀缓存），是大模型推理的核心优化技术：把请求里重复的前缀（静态内容）缓存下来，下次请求前缀完全一样时，直接复用，不用再算一遍，从而降延迟、省成本、减算力。

## 什么是上下文

LLM是没有记忆的，每次请求都是独立的，因此，每次对话都必须带上完整的上下文，才能得到准确的回答。我们在调用大模型接口时，传的messages就是完整的上下文，大模型的记忆完全靠我们每次塞给它的messages

messages里有三样：
- system 系统人设
- 每一轮用户提问
- 模型上一轮助手回答

模型是看着「用户问啥 + 它自己刚才答了啥」 接着往下聊的。

### 如果只发用户问题，不带模型历史回复，会发生啥？
举个例子：

第一轮
- 你：帮我写一个 Python 排序代码
- 模型：给了你一段完整代码

第二轮 你只发新问题，不带模型刚才的代码：
- 你：帮我把这段代码改成降序
- 👉 模型视角：我只看到你问了两句：
  - 1.帮我写 Python 排序代码
  - 2.帮我把这段代码改成降序

模型根本看不到它刚才给你的那段代码

它不知道「这段代码」是哪段，只能瞎猜、重写、答非所问。

常见后果
- 上下文断联：记不住刚才聊的内容
- 重复回答：反复重新开局，不接话
- 逻辑混乱：接不上上一轮的结论
- 改写、续写、迭代需求直接废掉

因此，在和大模型对话时，必须携带完整的上下文

## 大模型接口收到 messages 后，到底干了啥
LLM 接口收到 messages 后，按照以下几个步骤处理：拿到 messages → 按固定格式拼接成一整段长文本 → 切分成 Token → 进 Prefill 算 KV。

### 第一步：先搞懂核心前提
大模型**根本不认识JSON数组**，它只会读**一串连续的文字符号**。
你传的：
```json
[
  { "role": "system", "content": "你是专业程序员" },
  { "role": "user", "content": "解释一下KV缓存" },
  { "role": "assistant", "content": "KV缓存就是..." }
]
```
对模型来说就是**一堆角色+内容**，接口后台第一件事：**把数组压扁，拼成一段符合模型规则的连续文本**。

---

### 第二步：怎么把 messages 拼接成文本？
每个大模型都有**固定的角色分隔模板**（内置死规则），不是随便拼的。

#### 通用拼接逻辑（所有模型都一个套路）
给每个 `role` 加**专属标记头+尾**，把对话串起来：
1. 先写标记：`系统角色开始` + 系统内容
2. 再写标记：`用户开始` + 用户提问
3. 再写标记：`助手开始` + 助手回答
4. 最后停在：`助手开始`，等着模型往下生成

#### 举个直观模拟拼接
原始messages：
- system：你是专业程序员
- user：解释一下KV缓存
- assistant：KV缓存就是存注意力中间数据

后台自动拼成的**完整长文本**大概长这样（模拟格式）：
```
<|system|>
你是专业程序员
<|user|>
解释一下KV缓存
<|assistant|>
KV缓存就是存注意力中间数据
<|assistant|>
```
最后停在`<|assistant|>`后面，意思是：**该你模型接着往下说话了**。

> 关键点：
> 那些 `<|system|> <|user|> <|assistant|>` 都是**特殊占位符Token**，不是普通文字，模型专门用来区分谁在说话。

---

### 第三步：拼接完文本，怎么计算Token？
#### 1. 分词（Tokenize）
把上面拼好的**一整段长字符串**，按模型自带的**词表字典**切碎：
- 常见整词：`缓存` → 1个Token
- 生僻长词、英文、字母：拆成好几个Token
- 刚才的角色标记 `<|system|>` → 单独1个特殊Token

#### 2. 最终结果
一整段对话 + 角色标记，全部被切成**一长串数字ID列表**
> 文字 → 切碎成Token → 每个Token对应一个数字ID
大模型**不认识文字，只认识这串数字ID**。

#### 3. 计费、长度限制都按这个来
你看到的「消耗多少输入token」「上下文窗口4k/8k」，**算的就是这一整段拼完、切完的总Token数**。

---

### 第四步：Token之后怎么进入模型计算？
1. 把这一串Token数字，送入模型**Prefill预填充阶段**
2. 一次性并行计算所有Token的注意力，生成**全套KV缓存**
3. 有了KV状态，就开始**逐字Decode**，一个一个往外吐回答

---

### 串联一遍完整流程
1. 你前端构造 `messages` 对话数组
2. 接口后台按**角色模板**，把数组**拼接成带特殊标记的长文本**
3. 对长文本做**Token分词**，转成数字ID序列
4. 统计Token数量（计费、判超长）
5. 送入模型Prefill，计算生成KV缓存
6. 模型基于完整上下文，逐字生成回复

## 大模型回复时，是怎么生成文本的
前面提过，LLM核心就是一件事：根据当前输入，生成下一个字或者词，按理也是一段文本，那为什么LLM接口可以按照json的格式返回给我们，包括回复，工具调用等指令。

**实际上，模型在生成、续写、多轮对话全过程，自始至终都严格按一套固定规则拼接文本，和 messages 数组转文本是同一套规则。**

这里分两层讲解

---

### 一、先说结论
模型**不管是你发的 messages**，
还是**它自己正在一个字一个字生成回复**，
底层都遵循**同一套角色模板拼接规则**：
`<|系统|>、<|用户|>、<|助手|>` 这种特殊标记 + 内容拼接。

**不是随便凑文字，是按固定格式拼成一整条结构化长文本再预测下一个字。**

---

### 二、举实例，一眼看懂
#### 1. 你传的 messages 会被拼成：
```
<|system|>
你是智能助手
<|user|>
世界最高峰是
<|assistant|>
```
模型停在 `<|assistant|>` 后面，准备**从这里开始续写**。

#### 2. 模型开始逐字生成时
它每吐出一个字，**自动拼到整个结构化文本末尾**：

第一步生成「珠」→ 全局文本变成：
```
<|system|>
你是智能助手
<|user|>
世界最高峰是
<|assistant|>
珠
```

第二步再生成「穆」→ 自动继续往后拼：
```
<|system|>
你是智能助手
<|user|>
世界最高峰是
<|assistant|>
珠穆
```

**每生成一个 token，都原样套这套格式，追加到整段结构化文本最后。**

---

### 三、工具调用
普通对话拼接角色：
`<|system|> <|user|> <|assistant|>`

工具调用多两个角色标签：
`<|function_call|>`  助手要调用工具
`<|function_result|>` 工具返回结果给模型

#### 一、正常工具调用完整拼接全过程
##### 1. 第一层：System 先固定
里面写死：
你可以调用工具、参数格式、技能列表、入参规则。

##### 2. 用户提问
`<|user|> 今天北京天气怎么样？`

##### 3. 模型判断要调工具，开始生成 function_call
模型不会直接回答，而是在 `<|assistant|>` 后面，按格式生成工具调用结构：
```
<|assistant|>
<|function_call|>
{"name":"查天气","parameters":{"城市":"北京"}}
```
底层就是**普通文本续写**，只是续写的是JSON格式的工具调用内容。

##### 4. 后端拦截这个 function_call
接口后端识别到：模型输出了工具调用，**不再继续生成回答**，
而是去执行你的 Skill/函数/插件，拿到结果：
`北京：晴，25℃`

##### 5. 把工具结果按规则拼回去，塞给模型
后端构造一条**工具结果消息**，塞进 messages，按模板拼接：
```
<|function_result|>
北京：晴，25℃
```

##### 6. 模型再接着续写正常回答
模型看到 `<|function_result|>` 结果后，继续在后面生成自然语言：
```
<|assistant|>
北京今天天气晴朗，气温25度，适合出门。
```

---

#### 二、整体拼接后的完整长文本样子
给你看拼接完真实长文本结构，一眼看懂：
```
<|system|>
你有查天气、计算器等工具，严格按JSON格式调用
<|user|>
今天北京天气怎么样？
<|assistant|>
<|function_call|>
{"name":"查天气","parameters":{"城市":"北京"}}
<|function_result|>
北京：晴，25℃
<|assistant|>
北京今天天气晴朗，气温25度，适合出门。
```

**全程就是一套拼接逻辑，只是多加了两个标签。**

---

#### 三、和普通对话最大区别
1. 普通对话：只循环 user ↔ assistant
2. 工具调用：多了一轮闭环
`user → assistant(发工具调用) → function_result → assistant(给答案)`

3. 所有工具调用、Skill调用、插件调用：
**底层没有特殊魔法，依然是模板拼接+Token预测+KV缓存**
模型只是**“按格式续写一段工具调用文本”**，后端识别后执行，再把结果拼回上下文。

---

---

### 四、关键核心：为什么必须按规则拼接？
1. 模型**天生靠格式标记区分谁说话**
- `<|user|>` 后面是用户说的
- `<|assistant|>` 后面是模型要说的
2. 它续写时**永远只在 <|assistant|> 标签后面往下接字**
绝不会跑到用户位置、系统位置乱生成。
3. 整套格式一旦乱了，模型就会：
乱插话、自己冒充用户、逻辑跑偏、答非所问。

---

### 五、关联之前所有知识点
1. messages 数组 → 按固定模板**拼接结构化长文本**
2. 转 Token → Prefill → 生成 KV 缓存
3. 模型逐字生成时，**依然沿用同一套拼接规则**，不断往后追加文字
4. 追加的每一个字，只做**增量 KV 更新**，不用从头重算
5. Prompt Caching 缓存的，就是**这套拼好的完整结构化前缀**

---

### 六、一句终极总结
**整个从调用接口、多轮对话、模型逐字生成，全程从头到尾，
都是按同一套固定角色格式拼接成一条长文本，
模型只认这条格式化后的长文本，不认裸文字、不认JSON数组。**

最后，LLM接口根据这些标签解析生成json格式结果

## 什么是提示词缓存
**Prompt Caching（提示词缓存/前缀缓存）**，是大模型推理的核心优化技术：**把请求里重复的前缀（静态内容）缓存下来，下次请求前缀完全一样时，直接复用，不用再算一遍**，从而**降延迟、省成本、减算力**。

### 一、核心原理
- 大模型推理分两阶段：
  - **Prefill（预填充）**：把提示词转成Token、算注意力KV状态，**计算最重、最费钱**。
  - **Decode（解码）**：在已有上下文上逐字生成，成本低。
- Prompt Caching：把**Prefill算出的KV Cache跨请求暂存**；新请求若**前缀完全一致（Token级精确匹配）**，直接复用缓存，**跳过Prefill，只算新增部分**。

### 二、能带来什么好处
- ⚡ **延迟大减**：长提示词场景，响应可从秒级降到**百毫秒级**。
- 💰 **成本大降**：缓存命中的输入Token计费通常是原价的**1/10~1/4**（如OpenAI/Anthropic新模型）。
- 🧠 **算力节省**：避免重复的Transformer注意力计算，GPU占用大幅降低。

### 三、生效规则（关键）
- **前缀必须精确匹配**：任何Token差异（空格、标点、大小写）都会失效。
- **长度门槛**：如OpenAI要求前缀≥**1024 Token**，按**128 Token**增量对齐。
- **缓存时效**：通常**5~10分钟**无访问自动过期，低峰最长约**1小时**。
- **最佳写法**：**静态内容（系统指令、角色、示例）放前面，动态内容（用户输入、变量）放后面**。

### 四、典型适用场景
- 多轮对话（上下文复用）
- 固定系统指令+动态用户问题
- 反复处理同一份长文档/知识库
- 高频模板化查询（客服、FAQ、RAG）

一句话总结：**Prompt Caching就是“同样的前缀只算一次”，让长提示词请求又快又便宜**。
